## The FFN is where the parameters live. What if not all of it has to run?

The feed-forward lesson made a small observation that quietly reshapes modern model design: most of a transformer's parameters live in the FFN, not the attention. At $d_{\text{model}} = 4096$ with a 4× expansion, each FFN is $\sim$33M parameters; across 32 blocks that's a billion parameters in FFNs alone, dwarfing the $\sim$200M in attention projections.

That asymmetry creates an opening. What if you had *more* FFN — say, eight different FFNs sitting side by side, each specialized for different kinds of tokens — but only ran one or two of them per token? You'd carry the parameters but skip most of the compute. Model capacity would grow 8×; compute would only grow 2×. The model would have eight different "thinking styles" available; each token would only use the most relevant two.

This is the **mixture of experts (MoE)** trick, and it's the architectural choice behind Mixtral-8×7B, DeepSeek-V2/V3, GPT-OSS variants, and Switch Transformer. The compute story is simple: more capacity at the same FLOPs. The training story is harder, because you need a *router* that picks the right experts for each token, and the router has to be learned from gradients alone.

## The trick

Replace one big FFN with $N$ smaller "expert" FFNs $E_1, \ldots, E_N$ and a small router $g$. For each token's hidden vector $\mathbf{x}$, the router produces $N$ logits, softmaxes them, picks the top-$k$ experts, and renormalizes their weights. The output is the weighted sum of the chosen experts' outputs.

$$
\text{MoE}(\mathbf{x}) = \sum_{i \in \text{top-}k(g(\mathbf{x}))} w_i \cdot E_i(\mathbf{x})
$$

The expert *not* in the top-$k$ doesn't run at all — its FLOPs are skipped. That's where the compute savings come from. The expert parameters still exist in memory; they're just dormant for this token.

<MathCode
  math="\text{MoE}(\mathbf{x}) = \sum_{i \in \text{top-}k(g(\mathbf{x}))} w_i \cdot E_i(\mathbf{x})"
  code={`# Mixture of experts: one small router + N expert FFNs.
class MoE(nn.Module):
    def __init__(self, d_model, d_ff, n_experts, top_k):
        super().__init__()
        self.experts = nn.ModuleList(
            [FeedForward(d_model, d_ff) for _ in range(n_experts)]
        )
        self.router = nn.Linear(d_model, n_experts, bias=False)
        self.top_k = top_k

    def forward(self, x):
        logits = self.router(x)               # [B, T, n_experts]
        weights, indices = topk(softmax(logits), self.top_k)
        weights = weights / weights.sum(-1, keepdim=True)   # renormalize

        out = torch.zeros_like(x)
        for i in range(self.top_k):
            for e in range(len(self.experts)):
                mask = (indices[..., i] == e)
                if mask.any():
                    out[mask] += weights[..., i:i+1][mask] * self.experts[e](x[mask])
        return out`}

The load-balancing problem is real: if the router consistently picks the same few experts, the rest never get gradient and the model collapses to a smaller dense one. Production training adds an **auxiliary load-balancing loss** that penalizes uneven utilization, keeping the per-expert load near uniform.

<Callout title="Why the aux loss matters" targetInteractive="moe-router">
  Imagine the model never updates the lopsided routing. Idle experts get no gradient through this layer (their output is zero-weighted, so $\partial L / \partial E_i = 0$), so they stay random forever. Production training drives the imbalance number toward 1.0 by adding a penalty term: `L += alpha * sum_e (load_e^2)`. The model has to *want* to use every expert, even when it doesn't.
</Callout>

## Reading the result

Three things to internalize.

**MoE decouples parameter count from active FLOPs.** A dense model has params ≈ FLOPs (every parameter is touched per token). MoE has params = N × FLOPs / k. This is the only architectural lever that lets you scale model capacity faster than you scale compute, which is why MoE became the default for frontier open-weights models once load-balancing was solved well enough.

**The router is a learned softmax over experts.** Just a small linear projection followed by top-k. No magic. What makes it work is the auxiliary load-balancing loss that forces the router to spread tokens across experts; without it, the router collapses to "always pick the same few experts" and you've trained a smaller dense model with extra parameters wasted.

**Small-N, small-k is the dominant choice for the "few big experts" branch.** Mixtral-8×7B uses top-2 of 8; DBRX uses top-4 of 16; Qwen MoE variants land in the same neighborhood. The other branch is "many tiny experts": DeepSeek-V2 uses top-6 of 160 (plus 2 shared experts always active); Switch Transformer used top-1 of up to 128. Both branches work; the small-N branch is simpler to train and ship, the many-tiny branch trades complexity for finer-grained specialization.

## Recap

Three takeaways. **Replace one FFN with N expert FFNs and a router that picks top-k per token.** Compute scales as k; parameters scale as N. **A learned softmax over N experts plus an auxiliary load-balancing loss is the recipe.** Without the aux loss, training collapses; with it, every expert eventually gets used. **Top-2 of 8 is what production picks** — the sweet spot between active-FLOPs efficiency and trainability.
