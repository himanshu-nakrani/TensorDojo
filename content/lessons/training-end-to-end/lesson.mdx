## You have a model, a loss, an optimizer, a schedule, a dataset. What does one training run actually look like?

The previous lessons built each piece individually. **Backprop** computes the gradient of a loss with respect to every parameter in a model. **SGD** samples batches of examples and averages their gradients. **Optimizers** turn gradients into parameter updates. **Schedules** scale the update size over time. **Mixed precision** and **gradient checkpointing** make the whole thing fit on real hardware. The training *run* is the loop that wires them all together.

The lesson is the loop. For step in range(N): sample a batch → forward → loss → backward → optimizer step → schedule the LR → log. Every step is one of the pieces from a previous lesson. The thing the reader gets to *see* is the loss curve over thousands of steps, with all the pieces wired up — and a few presets that show what each piece contributes.

## The loop

$$
\text{for } t = 0, 1, \ldots, T-1: \quad
\theta \leftarrow \theta - \eta(t) \cdot \text{step}(\theta, g_t, \text{state}_t)
$$

where $g_t$ is the gradient of the loss over the batch sampled at step $t$, and the per-step $\text{step}$ is whatever optimizer the run is using (SGD, momentum, Adam).

## Try it

The centerpiece is a 2D 3-class classification task (~200 samples, ~67 per class, labels assigned by which of three angular sectors the point falls in). The model is a tiny MLP: 2 → 8 → 8 → 3 with ReLU activations, 123 parameters (16 + 8 + 64 + 8 + 24 + 3, weights and biases for the three layers). The default config trains to ≥ 90% test accuracy in 300 steps. Three preset buttons — Default, LR too high, No schedule — reconfigure all the controls and let you re-run with a single click.

## Reading the result

**A training run is a loop.** The forward, loss, backward, optimizer step, and LR scheduling are all operations the loop calls once per step. The math of each operation is from a previous lesson; the *composition* is what training is. Once you can read the loop, you can read any modern training script — they all reduce to this shape, with different model architectures, different loss functions, different optimizers, and different schedules. The differences are in the *pieces*, not the *loop*.

**The three presets isolate the three knobs.** "Default" uses Adam + warmup+cosine + LR 5e-3. "LR too high" uses SGD + constant + LR 5.0. "No schedule" uses Adam + constant + LR 5e-3. The difference between default and diverges is *mostly* the LR peak and the optimizer; the difference between default and no-schedule is *only* the schedule. The comparison widget runs all three on the same dataset and shows the difference side by side, with no other variables changed. This is how you debug a real training run: change one knob at a time.

**The default config is the conventional modern recipe for a reason.** Adam + warmup + cosine + a small batch is robust across a wide range of problems. You don't tune the optimizer or the schedule shape; you tune the peak LR. Most of the engineering effort in modern training is in the *data pipeline* — the sampler, the augmentation, the shuffling — not in the loop itself. The loop is solved; the data is the problem.

## Recap

Three takeaways. **A training run is a loop** that calls the previous four lessons' pieces in sequence: sample a batch, forward, loss, backward, optimizer step, schedule the LR, log. The pieces are independent of each other; the loop is what composes them. **The default config — Adam + warmup + cosine + a small batch — is the conventional modern recipe** and a sensible starting point for almost any problem; tune the peak LR first, the rest later. **The three presets in the secondary widget isolate the three knobs** (optimizer, LR peak, schedule) by changing one at a time, which is how you debug a real training run. The loop runs the same way regardless of model size — but the very first question a real training run has to answer is "how big *should* the model be?" The next lesson is **scaling laws**: the empirical curves that tell you how to split a fixed compute budget between parameters and training tokens.
